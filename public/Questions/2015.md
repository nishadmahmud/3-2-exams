### 1(c)
State the advantage and disadvantage of water fall model. **(3)**
> **Advantage:** Simple and easy to understand. Phases are rigid and do not overlap, making project tracking and management straightforward.
> **Disadvantage:** Highly inflexible. It is extremely difficult to accommodate changing requirements once a phase is completed, and working software is produced very late in the cycle.

### 1(d)
What is the main criterion for deciding whether or not to use the waterfall model in a software development project? **(2.5)**
> The main criterion is **Requirement Stability**. The waterfall model should only be used when the requirements are completely understood, unambiguous, and guaranteed not to change throughout the development process.

### 4(a)
What is a DFD? Discuss various components of a DFD. What do you understand by leveling of DFD? **(7)**
> **DFD (Data Flow Diagram):** A visual representation of how data flows through an information system, showing inputs, outputs, processes, and storage.
> 
> **Components:**
> 1. **Process (Circle/Rounded Rectangle):** Transforms incoming data flows into outgoing data flows.
> 2. **Data Store (Open Rectangle):** Represents data at rest (databases, files).
> 3. **External Entity (Square):** Sources or destinations of data outside the system boundary (e.g., a User).
> 4. **Data Flow (Arrow):** Represents the movement of data between components.
> 
> **Leveling of DFD:** 
> Leveling (or partitioning) is the process of breaking down a complex DFD into progressively detailed layers. It starts with a high-level overview (Level 0 / Context Diagram) and decomposes processes into sub-processes (Level 1, Level 2) to reveal detailed logic without overwhelming the viewer.

### 4(b)
What is Software Requirement Specification (SRS)? What are the characteristics of a good SRS document? **(6)**
> **SRS** is a formal document that comprehensively describes the expected behavior, constraints, and requirements of a proposed software system. It dictates *what* the system should do, without detailing *how* it will be implemented.
> 
> **Characteristics of a good SRS:**
> 1. **Correct:** Every requirement stated must actually be expected of the software.
> 2. **Unambiguous:** Every requirement has exactly one interpretation.
> 3. **Complete:** Includes all significant requirements and edge cases.
> 4. **Consistent:** No two requirements should contradict each other.
> 5. **Verifiable:** There must be a feasible process to test whether the requirement has been met.
> 6. **Traceable:** The origin of each requirement is clear and can be mapped to test cases.

### 4(c)
What are the difference between DFD and SCD (system Context Diagram)? **(4.5)**
> - A **System Context Diagram (SCD)** is essentially a "Level 0 DFD". It treats the entire software system as a single "black-box" process. It only shows how external entities interact with the system as a whole, omitting any internal logic, sub-processes, or internal databases.
> - A **DFD (Level 1 and above)** opens up that black box. It breaks the single system process down into multiple distinct sub-processes and explicitly shows the internal data stores (databases) and data flows between them.

### 5(b)
Define loop testing? Draw and describe different classes of loops. **(6.5)**
> **Loop Testing:** A white-box testing technique that focuses exclusively on the validity of loop constructs. It ensures loops execute the correct number of times and terminate properly.
> 
> **Classes of Loops:**
> 1. **Simple Loops:** A single standalone loop. Tested by skipping the loop entirely, executing exactly once, executing twice, executing $m$ times (where $m < n$), and executing for the maximum $n$, $n-1$, and $n+1$ times.
> 2. **Nested Loops:** A loop within a loop. Tested by starting at the innermost loop with all outer loops set to minimum values, conducting simple loop tests on the inner loop, and working outwards.
> 3. **Concatenated Loops:** Sequential loops. If independent, they are tested using simple loop testing. If dependent (e.g., the counter of the first is used in the second), they are treated as nested loops.
> 4. **Unstructured Loops:** Tangled, "spaghetti" logic loops (usually created by poorly placed `GOTO` statements). These should be redesigned rather than tested.

### 6(a)
Explain how the requirement process converts the client needs to validated Software Requirement Specification (SRS). **(6)**
> The Requirements Engineering process bridges the gap between client needs and the SRS through four main steps:
> 1. **Elicitation:** Engaging with clients (via interviews, surveys, observation) to discover their raw needs, business goals, and constraints.
> 2. **Analysis & Negotiation:** Analyzing the raw needs for feasibility. Conflicts between different stakeholders are negotiated and resolved, and requirements are prioritized.
> 3. **Specification:** The agreed-upon requirements are formally documented in the SRS using technical language, models, and precise use cases.
> 4. **Validation:** The drafted SRS is taken back to the client and development team for rigorous review to ensure it perfectly and realistically captures what needs to be built without errors.

### 6(b)
Compare the requirement analysis using structured analysis model and object oriented analysis model. **(5)**
> | Feature | Structured Analysis Model | Object-Oriented Analysis (OOA) |
> | :--- | :--- | :--- |
> | **Primary Focus** | Focuses on functions/processes and data as separate entities. | Focuses on "Objects" which encapsulate both data and functions together. |
> | **Approach** | Top-down approach (breaking complex processes down). | Bottom-up approach (building small objects and connecting them). |
> | **Modeling Tools** | DFDs (Data Flow Diagrams), ERDs (Entity Relationship Diagrams). | UML Diagrams (Use Cases, Class Diagrams, Sequence Diagrams). |
> | **Reusability** | Low reusability of code. | High reusability due to inheritance and polymorphism. |

### 6(c)
Discuss various checks and reviews in requirement validation. **(6.5)**
> During requirement validation, the SRS undergoes several critical checks:
> 1. **Validity Checks:** Does the system provide the exact functions that truly support the customer's business needs?
> 2. **Consistency Checks:** Are there any conflicting requirements? (e.g., one requirement asks for a feature, another forbids it).
> 3. **Completeness Checks:** Are all requested functions and constraints fully documented without missing details?
> 4. **Realism/Feasibility Checks:** Can these requirements actually be implemented given the current budget, timeline, and available technology?
> 5. **Verifiability Checks:** Is every requirement written in a way that can be objectively tested once the software is built?
