**(a)** Mention all the components of DFD with a proper diagram. **(2)**  
> **Components of a Data Flow Diagram (DFD):**
> 1. **Process (Circle or Rounded Rectangle):** Represents a function or task that transforms data inputs into outputs.
> 2. **Data Store (Open-ended Rectangle):** Represents a repository where data is stored at rest (e.g., a database or file).
> 3. **External Entity (Square or Rectangle):** Represents external actors (people, systems, or organizations) that send data to or receive data from the system.
> 4. **Data Flow (Arrow):** Shows the movement of data between entities, processes, and data stores.
> 
> ```mermaid
> graph LR
>   EE[External Entity] -->|Data Flow| P(Process)
>   P -->|Data Flow| DS[(Data Store)]
> ```

**(c)** Describe a real-world scenario where the Singleton design pattern is the best fit. Explain why this pattern is suitable in that context, and write the appropriate code to implement it. **(5)**
> **Scenario: Database Connection Pool**
> In an enterprise application, establishing a database connection is an expensive operation. If every client request created a new connection, the database would quickly become overwhelmed. The Singleton pattern ensures that only one instance of the Database Connection Pool exists across the entire application, centralizing connection management and preventing resource exhaustion.
> 
> **Java Code Implementation:**
> ```java
> public class DatabaseConnectionPool {
>     // 1. Create a private static instance
>     private static DatabaseConnectionPool instance;
> 
>     // 2. Make the constructor private
>     private DatabaseConnectionPool() {
>         // Initialize connection pool
>     }
> 
>     // 3. Provide a public static method to get the instance
>     public static DatabaseConnectionPool getInstance() {
>         if (instance == null) {
>             instance = new DatabaseConnectionPool();
>         }
>         return instance;
>     }
> }
> ```

**(c)** Imagine you are the manager of a digital Biryani outlet, which allows customers to customize their own Biryani orders. Suppose your outlet offers two types of base Biryani – Chicken Biryani and Beef Biryani. Customers can choose any one base. You also allow your customers to enhance their Biryani with various add-ons such as Boiled Egg, Extra Meat, and Salad, where each add-on increases the total cost. At any time, a customer should be able to check what they are ordering (description of the order) and the current total cost. Which design pattern would you follow to maintain the orders in your Biryani outlet? Write an appropriate code. **(5)**
> **Design Pattern:** The **Decorator Design Pattern**. This pattern allows behavior (add-ons) to be added to individual objects (base biryani) dynamically without affecting other objects, preventing an explosion of subclass combinations.
> 
> **Java Code:**
> ```java
> // Component Interface
> public interface Biryani {
>     String getDescription();
>     double getCost();
> }
> 
> // Concrete Components
> public class ChickenBiryani implements Biryani {
>     public String getDescription() { return "Chicken Biryani"; }
>     public double getCost() { return 250.0; }
> }
> 
> // Decorator Abstract Class
> public abstract class BiryaniDecorator implements Biryani {
>     protected Biryani tempBiryani;
>     public BiryaniDecorator(Biryani newBiryani) {
>         tempBiryani = newBiryani;
>     }
>     public String getDescription() { return tempBiryani.getDescription(); }
>     public double getCost() { return tempBiryani.getCost(); }
> }
> 
> // Concrete Decorator
> public class BoiledEgg extends BiryaniDecorator {
>     public BoiledEgg(Biryani newBiryani) { super(newBiryani); }
>     public String getDescription() { return tempBiryani.getDescription() + ", Boiled Egg"; }
>     public double getCost() { return tempBiryani.getCost() + 20.0; }
> }
> ```

**(a)** From the perspective of software engineering, explain the Open-Closed principle.
  * **i.** What measures should be taken to fulfill this principle?
  * **ii.** What are the potential consequences if this principle is violated? **(2)**  
> **Open-Closed Principle (OCP):** Software entities (classes, modules, functions) should be open for extension, but closed for modification.
> 
> **i. Measures to fulfill:** Rely heavily on Abstraction (using Interfaces or Abstract Classes) and Polymorphism. New functionality should be implemented by adding new classes that implement existing interfaces, rather than editing existing source code.
> 
> **ii. Consequences of violation:** Modifying existing code introduces a high risk of breaking previously working functionality. It leads to fragile codebases, tight coupling, and expensive testing cycles since old code must be re-tested every time a new feature is added.

**(a)** Your course teacher has assigned your group to prepare an SRS document... Explain what information should be included and excluded.  
> **Included in an SRS:**
> - Purpose and Scope of the system.
> - Overall Description (User characteristics, operating environment).
> - Functional Requirements (What the system must do).
> - Non-Functional Requirements (Performance, Security, Reliability).

> 
> **Excluded from an SRS:**
> - Internal design specifications (e.g., software architecture diagrams).
> - Implementation details (e.g., specific algorithms, coding languages).
> - Project management details (e.g., costs, schedules, budgets) unless acting as specific constraints.

**(b)** Differentiate between Top-down and Bottom-up integration.  
> **Top-Down Integration:** Testing begins with the top-level or main module. Lower-level modules that are not yet developed are simulated using *Stubs*. It verifies major control logic early but delays testing low-level utilities.
> 
> **Bottom-Up Integration:** Testing begins with the lowest-level modules. Higher-level modules that are not yet developed are simulated using *Drivers*. It verifies core utilities early but delays testing the overall system architecture.

> 2. **Integration Testing:** Ensuring that different modules work together (e.g., verifying that when a student registers, the fee payment module is correctly updated).

**(a)** Briefly explain the three key activities under risk assessment: risk identification, risk analysis, and risk prioritization. **(4)**
> 1. **Risk Identification:** The process of finding, recognizing, and describing potential risks that could affect the project. It involves brainstorming with stakeholders to identify what could go wrong (e.g., technical failures, staff turnover, budget cuts).
> 2. **Risk Analysis:** Assessing the identified risks to determine their nature, source, and potential impact. It involves understanding the likelihood of the risk occurring and the severity of its consequences on the project timeline, budget, or quality.
> 3. **Risk Prioritization:** Ranking the analyzed risks based on their combination of likelihood and impact (often using a risk matrix). High-probability, high-impact risks are prioritized for immediate mitigation planning, while low-priority risks are simply monitored.

**(b)** Write appropriate code for the following scenario:
You have to design a navigation system for a ride-hailing application that supports multiple travel modes: **Driving** and **Walking**, each with unique methods for calculating the travel time, and cost. Create a *NavigationContext* class that allows the user to set or change the travel mode dynamically during runtime, using a *TravelMode* interface with methods like *calculateTravelTime()*, and *estimateCost()*. Implement at least two concrete travel modes: **Driving** (consider fuel cost), **Walking** (assume zero cost). In the main class, show how the user can choose a travel mode and see the corresponding travel time and cost. **(6)**
> **Design Pattern:** Strategy Design Pattern
> 
> **Java Code Implementation:**
> ```java
> // 1. Strategy Interface
> public interface TravelMode {
>     double calculateTravelTime(double distanceInKm);
>     double estimateCost(double distanceInKm);
> }
> 
> // 2. Concrete Strategy: Driving
> public class DrivingMode implements TravelMode {
>     @Override
>     public double calculateTravelTime(double distanceInKm) {
>         // Assume average speed of 40 km/h
>         return (distanceInKm / 40.0) * 60; // in minutes
>     }
> 
>     @Override
>     public double estimateCost(double distanceInKm) {
>         // Fuel cost assumed at $0.50 per km + $2.00 base fare
>         return 2.0 + (distanceInKm * 0.50);
>     }
> }
> 
> // 3. Concrete Strategy: Walking
> public class WalkingMode implements TravelMode {
>     @Override
>     public double calculateTravelTime(double distanceInKm) {
>         // Assume walking speed of 5 km/h
>         return (distanceInKm / 5.0) * 60; // in minutes
>     }
> 
>     @Override
>     public double estimateCost(double distanceInKm) {
>         // Walking is free
>         return 0.0;
>     }
> }
> 
> // 4. Context Class
> public class NavigationContext {
>     private TravelMode travelMode;
> 
>     public void setTravelMode(TravelMode travelMode) {
>         this.travelMode = travelMode;
>     }
> 
>     public void displayRouteDetails(double distanceInKm) {
>         if (travelMode == null) {
>             System.out.println("Please set a travel mode first.");
>             return;
>         }
>         double time = travelMode.calculateTravelTime(distanceInKm);
>         double cost = travelMode.estimateCost(distanceInKm);
>         System.out.printf("Distance: %.1f km | Est. Time: %.1f mins | Est. Cost: $%.2f%n", 
>                           distanceInKm, time, cost);
>     }
> }
> 
> // 5. Main Class
> public class Main {
>     public static void main(String[] args) {
>         NavigationContext nav = new NavigationContext();
>         double distance = 10.0; // 10 km
> 
>         System.out.println("--- User selects Driving ---");
>         nav.setTravelMode(new DrivingMode());
>         nav.displayRouteDetails(distance);
> 
>         System.out.println("\\n--- User switches to Walking ---");
>         nav.setTravelMode(new WalkingMode());
>         nav.displayRouteDetails(distance);
>     }
> }
> ```
