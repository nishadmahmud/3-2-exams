### 1(a)
Differentiate between System software and Application software. **(2)**
> | Feature | System Software | Application Software |
> | :--- | :--- | :--- |
> | **Purpose** | Designed to manage system resources and provide a platform for other software to run. | Designed to fulfill a specific user requirement or perform a specific task. |
> | **Language** | Generally written in low-level languages (Assembly, C). | Written in high-level languages (Java, Python, C#). |
> | **Interaction** | Interacts directly with the hardware; runs in the background. | Interacts directly with the user; runs in the foreground. |
> | **Examples** | Operating Systems (Windows, Linux), Device Drivers. | Web Browsers, Word Processors, Games. |

### 1(b)
Suppose you want to test the food delivery application you developed for your project. What activities do you believe must be completed throughout the entire software testing life cycle? Explain briefly. **(3)**
> To test a food delivery application, the following Software Testing Life Cycle (STLC) activities must be completed:
> 1. **Requirement Analysis:** Understand what needs testing (e.g., location tracking, payment gateways, restaurant menus).
> 2. **Test Planning:** Determine the testing strategy, allocate resources, and select tools (e.g., Selenium for UI, JMeter for load testing).
> 3. **Test Case Development:** Write specific scenarios (e.g., "Add item to cart", "Simulate driver arriving", "Fail payment on invalid CVV").
> 4. **Environment Setup:** Configure the test servers, databases, and mock APIs (like a sandbox payment gateway).
> 5. **Test Execution:** Run the test cases on various devices (Android/iOS) and log any defects/bugs found.
> 6. **Test Cycle Closure:** Analyze test metrics, ensure all critical bugs are fixed, and sign off for release.

### 1(c)
Define Requirement Engineering and provide a brief overview of the steps involved in the process. **(5)**
> **Requirement Engineering** is the systematic process of defining, documenting, and maintaining the services that a customer requires from a system, along with the constraints under which it operates and is developed.
> 
> **Steps Involved:**
> 1. **Elicitation:** Gathering raw needs and constraints from stakeholders via interviews, surveys, or observation.
> 2. **Analysis & Negotiation:** Checking requirements for feasibility, resolving conflicts between different stakeholders, and prioritizing features.
> 3. **Specification:** Formally documenting the agreed-upon requirements into a Software Requirements Specification (SRS) document.
> 4. **Validation:** Reviewing the SRS with the client to ensure it accurately and completely captures their true needs.
> 5. **Management:** Managing changes to the requirements as the project progresses.

### 2(a)
Write the IEEE definition of software engineering. Explain the different phases of software system development. **(1+5)**
> **IEEE Definition:** "The application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software; that is, the application of engineering to software."
> 
> **Phases of Software System Development (SDLC):**
> 1. **Requirement Analysis:** Understanding and documenting what the system must do.
> 2. **System Design:** Architecting how the system will meet the requirements (database schemas, UI/UX, software architecture).
> 3. **Implementation (Coding):** Translating the design into executable source code.
> 4. **Testing:** Verifying that the code functions correctly and meets the initial requirements without bugs.

### 3(b)
Provide a brief description of the following terms: i. White box testing vs Black box testing ii. System testing vs Integration testing **(4)**
> **i. White box vs Black box testing:**
> - **White Box:** Focuses on the internal structure, logic, and code of the software. The tester knows how the code works and writes tests for specific paths and branches.
> - **Black Box:** Focuses solely on the inputs and outputs without any knowledge of the internal code. The tester treats the software as a "black box" and validates it against requirements.
> 
> **ii. System testing vs Integration testing:**
> - **Integration Testing:** Tests how individual modules or components work *together* to ensure their interfaces and data flows are correct.
> - **System Testing:** Tests the *entire, fully-integrated* application as a whole in an environment that closely mirrors production to verify it meets all overall requirements.

### 4(b)
Briefly elaborate the building components of SCRUM and explain how they are interconnected. **(3)**
> **Building Components of SCRUM:**
> - **Roles:** Product Owner (defines what to build), Scrum Master (facilitates the process), Development Team (builds the product).
> - **Artifacts:** Product Backlog (master list of all requirements), Sprint Backlog (tasks chosen for the current sprint), Increment (the working software produced).
> - **Events:** Sprint (timeboxed working period), Sprint Planning, Daily Standup, Sprint Review, Sprint Retrospective.
> 
> **Interconnection:**
> The **Product Owner** prioritizes the **Product Backlog**. During **Sprint Planning**, the **Team** selects items to form the **Sprint Backlog**. Over the course of the **Sprint**, the team holds **Daily Standups** to sync progress. By the end of the sprint, an **Increment** is produced, which is inspected during the **Sprint Review**, and the process itself is improved during the **Sprint Retrospective**.

### 5(c)
Discuss SOLID principles. Why should these principles be used? **(4)**
> **SOLID Principles:**
> - **S (Single Responsibility):** A class should have one, and only one, reason to change.
> - **O (Open/Closed):** Software entities should be open for extension but closed for modification.
> - **L (Liskov Substitution):** Subtypes must be substitutable for their base types without altering program correctness.
> - **I (Interface Segregation):** Clients should not be forced to depend upon interfaces they do not use (use many small, specific interfaces).
> - **D (Dependency Inversion):** High-level modules should depend on abstractions, not concrete implementations.
> 
> **Why use them?**
> Applying SOLID principles makes software architecture highly maintainable, scalable, and robust. It prevents "spaghetti code", makes unit testing much easier, and allows teams to add new features without breaking existing functionality.

> - **i. Portability:** The ability of software to run seamlessly across different environments, operating systems, or hardware architectures without requiring major modifications.

### 8(a)
What is Agile methodology? Discuss the advantages and disadvantages of Agile methodology. **(4)**
> **Agile Methodology** is an iterative and incremental approach to software development that emphasizes continuous delivery of working software, extreme adaptability to changing requirements, and tight collaboration between self-organizing teams and customers.
> 
> **Advantages:**
> - Highly flexible to changing client needs.
> - Delivers working software rapidly and continuously.
> - High customer satisfaction due to constant involvement.
> 
> **Disadvantages:**
> - Very difficult to accurately predict final project cost and timeline.
> - Can suffer from "scope creep" without strict management.
> - Places less emphasis on comprehensive documentation, which can hurt long-term maintenance.

### 8(b)
Imagine you are the owner of a digital ice-cream parlor, which allows customers to decorate their own ice-creams. Suppose your ice-parlor offers two flavors - Vanilla and Chocolate... customers can decorate with toppings (with additional cost)... Which design pattern would you follow? Write appropriate code. **(6)**
> The **Decorator Design Pattern** is perfect here. It allows us to dynamically attach additional toppings (and their costs) to a base ice-cream object at runtime without altering its structure.
> 
> ```java
> // Component Interface
> interface IceCream {
>     String getDescription();
>     double getCost();
> }
> 
> // Concrete Components (Base Flavors)
> class VanillaIceCream implements IceCream {
>     public String getDescription() { return "Vanilla Ice Cream"; }
>     public double getCost() { return 5.00; }
> }
> class ChocolateIceCream implements IceCream {
>     public String getDescription() { return "Chocolate Ice Cream"; }
>     public double getCost() { return 6.00; }
> }
> 
> // Base Decorator
> abstract class ToppingDecorator implements IceCream {
>     protected IceCream tempIceCream;
>     public ToppingDecorator(IceCream newIceCream) {
>         this.tempIceCream = newIceCream;
>     }
>     public String getDescription() { return tempIceCream.getDescription(); }
>     public double getCost() { return tempIceCream.getCost(); }
> }
> 
> // Concrete Decorators (Toppings)
> class ChocoChips extends ToppingDecorator {
>     public ChocoChips(IceCream newIceCream) { super(newIceCream); }
>     public String getDescription() { return tempIceCream.getDescription() + ", Choco Chips"; }
>     public double getCost() { return tempIceCream.getCost() + 1.50; }
> }
> class CashewNuts extends ToppingDecorator {
>     public CashewNuts(IceCream newIceCream) { super(newIceCream); }
>     public String getDescription() { return tempIceCream.getDescription() + ", Cashew Nuts"; }
>     public double getCost() { return tempIceCream.getCost() + 2.00; }
> }
> 
> // Main Application
> public class DigitalParlor {
>     public static void main(String[] args) {
>         // Customer orders Vanilla with Cashews and Choco Chips
>         IceCream myOrder = new ChocoChips(new CashewNuts(new VanillaIceCream()));
>         
>         System.out.println("Order: " + myOrder.getDescription());
>         System.out.println("Total Cost: $" + myOrder.getCost());
>         // Output:
>         // Order: Vanilla Ice Cream, Cashew Nuts, Choco Chips
>         // Total Cost: $8.50
>     }
> }
> ```
